Architecture Overview

BARO's backend is a FastAPI proxy layer sitting between the iOS client and the Korea Investment & Securities (KIS) Open API. The backend handles authentication, translates KIS responses into clean JSON for the mobile app, manages real-time WebSocket price streaming, and caches frequently-accessed data.

iOS App (SwiftUI) | |-- HTTPS REST ----> FastAPI Backend (port 8000) | | | |-- Auth Service ----> SQLite (baro_auth.db) | |-- KIS Client ------> KIS Open API (REST) | |-- KIS WS Client ---> KIS WebSocket (port 21000) | |-- WebSocket -----> FastAPI /ws/stream | |-- Stream Manager --> Fan-out to iOS clients |-- KIS WebSocket ---> Real-time ticks from KIS

Request Flow

1. Client Request → iOS app sends HTTP request with Bearer token

2. Auth Check → JWT validated via get_current_user() dependency (protected routes only)

3. Router → FastAPI router dispatches to appropriate service

4. Service Layer → Business logic, KIS API calls, caching

5. KIS Auth → OAuth token auto-refreshed if needed (5 min buffer)

6. KIS API → REST call to KIS with rate limiting (15 concurrent max)

7. Response Transform → KIS snake_case Korean fields → clean English JSON

8. Cache → Result cached (30s intraday, 5min daily, 60s market data)

Technical Decisions & Rationale

Every major technology choice explained — what was chosen, what alternatives were considered, and why the decision was made given BARO's specific requirements as a Korean financial investment platform.

Backend Framework: FastAPI (Python)

Decision

FastAPI 0.115.6 with Uvicorn as the async ASGI server.

Alternatives Considered

FrameworkLanguageStrengthsWhy Not Chosen
FastAPI Python Native async, auto-docs, Pydantic validation, WebSocket support, rapid prototyping CHOSEN
Django REST Framework Python Mature, large ecosystem, ORM, admin panel Synchronous by default; adding async/WebSocket requires channels + ASGI middleware. Heavy for a proxy layer. ORM unnecessary when KIS API is the primary data source.
Flask Python Lightweight, simple, huge community No native async support (requires gevent/eventlet). No built-in WebSocket. No automatic request validation or OpenAPI docs. Would need many extensions to match FastAPI's out-of-box features.
Express.js Node.js Fast I/O, huge npm ecosystem, JavaScript everywhere Python AI/ML ecosystem is critical for Phase 2+ (XGBoost, KoBERT, FinBERT-KO, TA-Lib, scikit-learn all have primary Python bindings). Using Node would require maintaining a separate Python service for ML inference, adding operational complexity.
NestJS TypeScript Structured, decorator-based, good for large teams Same Python AI ecosystem problem as Express. TypeScript adds safety but the ML model integration gap is a bigger concern.
Go (Gin/Fiber) Go Extreme performance, low memory, excellent concurrency No AI/ML library ecosystem. Would need Python sidecar for all ML inference. Go's WebSocket libraries are excellent but the team's Python expertise and AI requirements outweigh the raw performance advantage. BARO is I/O-bound (KIS API calls), not CPU-bound.
Spring Boot Java/Kotlin Enterprise-grade, strong typing, massive ecosystem Heavy startup time, verbose configuration. Python AI ecosystem unavailable. Over-engineered for a proxy layer that needs fast iteration.

Key Reasons for FastAPI

  1. Native async/await — BARO's backend is fundamentally an I/O proxy. It waits on KIS API responses, WebSocket streams, and database queries. Async is not optional, it's the core pattern. FastAPI + Uvicorn provide this natively with zero boilerplate.
  2. First-class WebSocket support — Real-time price streaming is a core feature. FastAPI handles WebSocket upgrade, connection lifecycle, and async message handling out of the box. No plugins or middleware required.
  3. Pydantic v2 validation — KIS API returns deeply nested Korean-language field names (e.g., stck_prpr, prdy_vrss). Pydantic models provide automatic parsing, validation, and transformation to clean English JSON — catching data issues at the boundary.
  4. Python AI/ML ecosystem — Phase 2+ requires XGBoost, KoBERT, FinBERT-KO, Isolation Forest, TA-Lib, BGE-M3, EXAONE 3.5, and more. All have primary Python bindings. Choosing Python now avoids a future rewrite or multi-language service split.
  5. Rapid prototyping — Auto-generated OpenAPI docs (/docs), automatic request/response serialization, and minimal boilerplate allow fast iteration on API design while maintaining type safety.
  6. Performance sufficient for use case — BARO is I/O-bound (network calls to KIS), not CPU-bound. FastAPI + Uvicorn handles thousands of concurrent connections efficiently. The bottleneck is KIS API rate limits (20 req/s), not framework throughput.

Decision

Korea Investment & Securities (한국투자증권) Open API as the primary market data provider, with a FastAPI proxy layer translating responses for the iOS client.

Key Reasons for KIS Open API

  1. Only viable option for real-time Korean + NXT data — BARO requires real-time quotes from both KRX (traditional exchange) and NXT (alternative trading venue). KIS is one of the few providers offering unified access to both venues through a single API.
  2. WebSocket real-time streaming — KIS provides persistent WebSocket connections with sub-second trade tick delivery. Essential for live price updates in the stock detail screen.
  3. Future order execution — KIS API supports full order lifecycle (place, modify, cancel). When BARO adds trading features, no provider migration needed.
  4. Rich data depth — Order book (호가), investor flow by type (foreign/institutional/individual), rankings (volume, market cap, gainers/losers), and US market data — all from one provider.
  5. Free for brokerage account holders — No per-call fees or monthly subscription for data access. Cost scales with infrastructure, not data consumption.
  6. Proxy architecture decision — KIS API requires appkey/appsecret credentials that cannot be embedded in the iOS app. The FastAPI backend acts as a secure proxy, keeping credentials server-side while translating KIS's Korean field names into clean English JSON for the mobile client.

iOS Architecture: SwiftUI + MVVM + Repository

Decision

SwiftUI with @Observable macro, MVVM pattern, protocol-based Repository layer, and DependencyContainer for injection.

Alternatives Considered

ChoiceStrengthsWhy Not Chosen
SwiftUI + @Observable Declarative, reactive, minimal boilerplate, Swift 6 native CHOSEN
UIKit + Storyboards Mature, full control, huge legacy codebase support Imperative UI requires significantly more code for the same result. Manual layout constraints are error-prone. SwiftUI's declarative approach matches the data-heavy, list-rich nature of a financial app perfectly.
UIKit + Programmatic No storyboard merge conflicts, full control Still imperative. Requires manual state synchronization between data and views. The stock detail screen alone has 17 sub-components — managing their state imperatively would be fragile.
React Native Cross-platform, JavaScript, hot reload Performance concerns for real-time WebSocket + Canvas chart rendering. The stock chart (StockChartView) requires 60fps drawing with zoom/pan/scrubbing — native Canvas outperforms React Native bridge significantly. Also loses access to native iOS features like Keychain, Apple Sign In, and Combine.
Flutter Cross-platform, excellent performance, Dart Good performance but Dart ecosystem is smaller for financial libraries. No native Keychain/Apple Sign In integration without plugins. The team's Swift expertise and iOS-first strategy favor native development. Custom chart rendering in Flutter's Skia is powerful but SwiftUI Canvas is simpler for this use case.
SwiftUI + TCA (Composable Architecture) Unidirectional data flow, excellent testability, reducer pattern Higher learning curve and more boilerplate than @Observable. TCA excels for complex state machines but BARO's screen-level state is straightforward (load data, display, update via WebSocket). @Observable provides sufficient reactivity with less ceremony.

Key Reasons

  1. @Observable macro (Swift 6) — Automatic view updates when data changes, zero boilerplate. ViewModels just need @Observable annotation. No @Published wrappers, no ObservableObject conformance, no objectWillChange publishers.
  2. Declarative lists for financial data — Stock ranking lists, news feeds, order books, and transaction histories are all list-heavy UIs. SwiftUI's List/LazyVStack with automatic diffing handles these efficiently.
  3. Native Canvas for charts — StockChartView renders OHLCV candles, technical overlays (Bollinger, Ichimoku, MA), and sub-indicators (RSI, MACD) at 60fps with gesture-based zoom/pan. Native Canvas provides the performance needed.
  4. Protocol-based Repository pattern — MarketRepositoryProtocol allows swapping between APIMarketRepository (live KIS) and MockMarketRepository (development) via DependencyContainer(useMocks: true/false). Critical for development velocity when KIS API has rate limits.
  5. Native Keychain + Apple Sign In — Secure token storage via iOS Keychain with kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly. Apple Sign In via AuthenticationServices framework. Both require native Swift integration.
  6. Combine for WebSocket — WebSocketManager uses Combine publishers and AsyncStream for per-stock price update subscriptions. Native integration with URLSession WebSocket task.

Real-Time Strategy: WebSocket Relay

Decision

Backend maintains a persistent WebSocket to KIS, parses binary/pipe-delimited messages, and fans out clean JSON to iOS clients via a second WebSocket layer.

Alternatives Considered

ApproachHow It WorksWhy Not Chosen
WebSocket relay (proxy) Backend ↔ KIS WS, Backend ↔ iOS WS CHOSEN
Direct iOS ↔ KIS WebSocket iOS connects directly to KIS WebSocket KIS WebSocket requires appkey/appsecret in the connection handshake — embedding credentials in the iOS binary is a critical security risk. KIS sends pipe-delimited binary data that requires server-side parsing. Each iOS client would need its own KIS connection (doesn't scale). No fan-out or subscription sharing.
HTTP polling iOS polls REST endpoints every N seconds Unacceptable latency for live price updates. 1-second polling × 20 stocks = 20 req/s per user, hitting KIS rate limits immediately. Wastes battery and bandwidth. No way to achieve sub-second price display.
Server-Sent Events (SSE) Server pushes events over HTTP Unidirectional only (server → client). iOS cannot send subscribe/unsubscribe messages back. Would need a separate REST endpoint for subscription management, adding complexity. WebSocket provides bidirectional communication natively.
Firebase Realtime Database Write prices to Firebase, iOS subscribes Adds latency (KIS → Backend → Firebase → iOS instead of KIS → Backend → iOS). Per-connection pricing at scale. Vendor dependency for a critical data path. Firebase is designed for client-to-client sync, not server-push market data.

Key Reasons for WebSocket Relay

  1. Credential isolation — KIS appkey/appsecret never leave the backend. iOS clients authenticate with JWT tokens only.
  2. Connection fan-out — One KIS WebSocket connection serves all iOS clients. If 100 users watch Samsung (005930), only 1 KIS subscription is active. StreamManager tracks reference counts and auto-unsubscribes when the last client disconnects.
  3. Data transformation — KIS sends pipe-delimited (|) messages with caret-separated (^) fields in Korean encoding. Backend parses these into clean JSON that iOS can directly deserialize into Swift models.
  4. Venue abstraction — Backend handles KRX/NXT/Combined venue routing (different KIS TR IDs per venue). iOS simply sends {"venue": "combined"} and backend subscribes to the right KIS feed.

AI Infrastructure Choices

Knowledge Retrieval: CAG + RAG + Local LLM (3-tier)

ApproachHowWhy This Over Alternatives
CAG (Cache-Augmented) Pre-cache stable content (regulations, prompts) in Claude's KV cache 90% cost reduction on system prompts. 0ms retrieval. FSC regulations and MCP tool schemas rarely change — perfect for caching. Alternative (embedding in prompt every time) wastes $49K+/month at 10K MAU.
RAG (Retrieval-Augmented) Hybrid search (vector + BM25) on pgvector Financial documents are too large and numerous to fit in context. RAG retrieves only the relevant 2,000 tokens. Alternative (fine-tuning on all documents) is expensive, slow to update, and can't handle access controls per user.
Local LLM QLoRA fine-tuned EXAONE for simple queries 87% cost reduction at scale by handling simple Korean Q&A locally. Alternative (all-cloud) costs $5,000/month. Confidence threshold (0.75) ensures quality by falling back to Claude for uncertain queries.

Reranking: Cohere Rerank API

OptionCostWhy Not Chosen
Cohere Rerank $1/1,000 calls CHOSEN — "Largest quality improvement in the entire RAG pipeline." Reranking is the single highest-ROI step — ensures the 2,000 retrieved tokens contain the best possible information for Claude.
BGE Reranker v2 $0 (local) Kept as fallback — Slightly lower quality but free. Used when Cohere is unavailable or for cost-sensitive queries.
No reranking $0 Significantly degrades answer quality. Vector search retrieves "related" chunks, but reranking selects "most useful" chunks. The difference is measurable in financial accuracy.
LLM-based reranking High (Claude tokens) Too expensive at scale. Sending all candidate chunks to Claude for ranking negates the cost savings of the entire RAG pipeline.

Hybrid Search: Dense Vector (50%) + BM25 Keyword (30%) + Metadata (filter)

Why hybrid over pure vector search?

  • Financial queries have exact terms — Users search for "삼성전자 2024 Q3 영업이익" (Samsung Electronics 2024 Q3 operating profit). BM25 excels at exact term matching that vector search may miss.
  • Stock codes are not semantic — "005930" has no semantic meaning to an embedding model, but BM25 matches it exactly.
  • Reciprocal Rank Fusion — Combines both ranking lists without needing to tune weights manually. Documents that rank highly in both semantic relevance and keyword matching are promoted.
  • Metadata filtering applied first — Eliminates irrelevant documents (wrong language, wrong access level, wrong date range) before expensive similarity computation.

Orchestration: LangGraph + MCP

Agent Orchestration: LangGraph

FrameworkStrengthsWhy Not Chosen
LangGraph Typed state graphs, conditional edges, checkpointing, native LangChain integration CHOSEN
LangChain Agents (LCEL) Simple, well-documented, large community Sequential chains can't handle BARO's branching workflows. Risk Guard has 3 conditional branches (approve/monitor/freeze). Trade Signal has HITL wait states. LangChain agents lack native support for conditional branching, loops, and human-in-the-loop checkpoints.
AutoGen (Microsoft) Multi-agent conversations, code execution Conversation-based agent coordination is unpredictable for financial workflows. BARO needs deterministic state machines, not open-ended agent chat. AutoGen's "agents talking to each other" pattern violates the inter-agent firewall requirement (no free-form text between agents).
CrewAI Role-based agents, task delegation Similar issues to AutoGen — relies on natural language task delegation. BARO's compliance requirements demand typed state transitions with JSON Schema validation between agents, not natural language instructions.
Custom state machine Full control, no dependencies Reinventing checkpointing (server restart recovery), typed state graphs, and conditional edges is significant engineering effort. LangGraph provides these out of the box and is actively maintained.
Temporal / Prefect Production workflow engines, excellent reliability Designed for ETL/data pipelines, not LLM-native workflows. No built-in LLM context management, prompt chaining, or tool use patterns. Would need extensive adaptation for AI agent use cases.

Key Reasons for LangGraph

  1. Checkpointing — "Like a video game save point." If the server restarts during a 7-day HITL wait for Portfolio Optimizer approval, no progress is lost. The graph resumes from the last checkpoint.
  2. Typed state — Each agent carries a typed state clipboard through its workflow. The compiler catches type errors in agent-to-agent data passing, not runtime.
  3. Conditional edges — Risk Guard's anomaly score branches (score <0.3 → approve, 0.3-0.7 → monitor, >0.7 → freeze) are first-class graph constructs, not if/else spaghetti.
  4. HITL as a graph node — Human approval wait states are native graph nodes with configurable timeouts. Not a hack bolted onto a sequential chain.

Tool Access: MCP (Model Context Protocol)

ApproachWhy Not Chosen
MCP Servers (6 servers, 38 tools) CHOSEN — Standardized tool interface. Each server defines its own schema, authentication, and rate limits. Agents can only call tools in servers they have access to. New tools can be added without changing agent code.
Direct function calls No access control boundary between agents and tools. Any agent could call any function including dangerous ones (e.g., direct DB writes). MCP servers enforce per-agent tool authorization.
OpenAI function calling Vendor-specific. BARO uses Claude as primary LLM. MCP is model-agnostic and works with any LLM that supports tool use.
Custom tool registry Reinventing MCP's authentication, schema validation, and tool discovery. MCP is becoming an industry standard with community tooling.

Job Queue: BullMQ

QueueStrengthsWhy Not Chosen
BullMQ Redis-backed, persistent, retries, delayed jobs, priorities CHOSEN — Already using Redis (Upstash). No new infrastructure. Supports job priorities (compliance checks before notifications), scheduled jobs (daily market analysis), and retry with backoff (KIS API failures).
Celery Python-native, mature, flexible backends Heavier setup, more dependencies. BullMQ is simpler and uses the same Redis instance already provisioned for caching and pub/sub.
AWS SQS Managed, highly durable Vendor lock-in, no Redis pub/sub integration, per-message pricing. BARO's infrastructure is on Railway/Supabase, not AWS.

Deployment: Railway Seoul + Vercel + Cloudflare

ServiceRoleWhy ChosenAlternatives Rejected
Railway (Seoul) Backend containers, GPU inference Seoul data center provides <10ms latency to KIS API servers (also in Korea). L4 GPU instances available for local LLM serving. Simple container deployment. No Kubernetes complexity. AWS Seoul (ap-northeast-2): More powerful but significantly more complex and expensive for a startup. Kubernetes/ECS overhead. GCP: No Seoul region. Heroku: No Seoul region, no GPU instances.

AWS CloudFront: Vendor lock-in with AWS. No CDN: Unacceptable for a financial app with real-time data and security requirements.

Why Seoul Region is Non-Negotiable

  1. KIS API latency — KIS servers are in Korea. Every REST call and WebSocket message traverses the network. Seoul region gives <10ms round-trip vs 100ms+ from US/EU regions. For real-time price streaming, this difference is visible to users.
  2. PIPA compliance — Korean Personal Information Protection Act (개인정보보호법) requires personal data of Korean citizens to be stored and processed within Korea or in approved jurisdictions. Seoul hosting simplifies compliance.
  3. FSC regulatory alignment — Korean Financial Services Commission guidelines expect financial services to maintain infrastructure accessible for regulatory inspection. Domestic hosting satisfies this requirement.
  4. User base — BARO's primary users are Korean investors. Sub-50ms API response times are achievable only with Korea-based infrastructure.

Real-Time Data (WebSocket)

Price Streaming WebSocketManager → StreamManager
Live WebSocket

iOS: WebSocketManager connects to ws://host:8000/ws/stream

Backend: StreamManager → KISWebSocket → KIS WS (port 21000)

Flow:

  1. iOS sends: {"action": "subscribe", "codes": ["005930"], "venue": "combined"}
  2. StreamManager registers client subscription
  3. If first subscriber for this (code, venue), subscribes to KIS WebSocket
  4. KIS sends pipe-delimited trade ticks
  5. Backend parses to JSON: {"type": "trade", "code": "005930", "price": 72300, ...}
  6. StreamManager fans out to all subscribed iOS clients

KIS WS TR IDs:
KRX trades: H0STCNT0 | KRX orderbook: H0STASP0
NXT trades: H0NXCNT0 | NXT orderbook: H0NXASP0
Combined trades: H0UNCNT0 | Combined orderbook: H0UNASP0
Auto-reconnect with exponential backoff (max 60s). PINGPONG heartbeat. Per-code AsyncStream on iOS side.

AI Architecture Overview

BARO's AI platform is a 7-layer architecture with 12 specialized agents orchestrated via LangGraph, using a dual-LLM strategy (Claude cloud + EXAONE 3.5 local), with 6 MCP servers providing 38 structured tools.

Layer 7: LangGraph ── Typed state graphs, conditional edges, checkpointing Layer 6: MCP Servers (6) ── 38 tools: trading, portfolio, market, compliance, notifications, documents Layer 5: Application ── Next.js 14, React Native, TradingView Charts Layer 4: Orchestration ── LangGraph, BullMQ, Redis Pub/Sub Layer 3: Data ── KIS API, TimescaleDB, pgvector, Redis, TA-Lib, Naver Finance, OANDA, Upbit Layer 2: Models ── Claude, XGBoost, KoBERT, FinBERT-KO, BGE-M3, EXAONE 3.5, Isolation Forest, LSTM Layer 1: Infrastructure ── Vercel, Railway Seoul, Supabase, Upstash Redis, Cloudflare

1. One Agent = One Job — Every agent has exactly one responsibility. No god agents.

2. HITL on Every Money Action — A human must approve before any financial transaction executes.

3. Immutable Audit at Every Terminal Node — Every time an agent completes a workflow, an immutable audit record is written.

Stages 2–8 — Processing Pipeline

Stage 2 — Conversion & OCR: Confidence threshold 0.85 minimum. Below threshold: quarantine for human review.

Stage 3 — Semantic Chunking: 200–400 tokens per chunk, 50 token overlap. Metadata: source_doc_id, page_number, section_heading, modality, language, access_level.

Stage 4 — Embedding: Primary: BGE-M3 (1024 dimensions, $0 local). Fallback: text-embedding-3-small ($0.02/1M tokens). Phase 3: CLIP ViT-L/14 (image), ImageBind (cross-modal).

Stage 5 — Hybrid Retrieval: Dense vector search (50% weight, cosine similarity) + BM25 keyword search (30% weight) + metadata filtering (applied first). Combined via Reciprocal Rank Fusion (RRF).

Stage 6 — Reranking: Primary: Cohere Rerank API ($1/1K calls). Fallback: BGE Reranker v2 ($0 local). Largest quality improvement in the pipeline.

Stage 7 — Permission Filtering: Hard gate — remove unauthorized chunks before Claude sees them. Row-Level Security enforced at Supabase layer.

Stage 8 — Context Assembly: ~3,600 token budget combining system prompt (CAG), query, retrieved chunks, live data, user profile, and conversation history.

  2. CAG — Pre-loaded stable knowledge (faster, cheaper)
  3. Hybrid Search — Semantic + keyword matching
  4. Reranking — Cohere reranker for precision
  5. Parallel Context Fetching — Fetch multiple sources simultaneously
  6. Permission-Filtered Context — Remove unauthorized chunks before model sees them
  7. Time-Windowed Context — Only include recent data
